# opencascade-rs → truck 移植案

作成: 2026-03-02

---

## 1. 現状の opencascade-rs 依存箇所の整理

| ファイル | 使用している OCC API |
|---|---|
| `step_to_brep.rs` | `Shape::read_step()`, `shape.write_brep_bin()` |
| `shape.rs` | `Shape::read_brep_text()`, `Shape::read_brep_bin()`, `shape.mesh_with_tolerance()`, `shape.edges()` |
| `shape_to_glb.rs` | `opencascade::mesh::Mesh`（頂点・インデックス）, `edge.approximation_segments()` |
| `shape_stretch.rs` | `Shape::half_space()`, `shape.intersect()`, `shape.subtract()`, `shape.union()`, `shape.set_global_translation()`, `shape.clean()`, `shape.deep_copy()`, `face.normal_at_center()`, `face.center_of_mass()`, `face.extrude()`, `Shape::box_from_corners()`, `Shape::cylinder()`, `Shape::empty()` |
| テストコード | 上記全般 + `Shape::read_step()` |

## 2. truck クレート構成と対応関係

```toml
[dependencies]
truck-modeling  = "0.4"   # builder API、変換関数
truck-topology  = "0.6"   # Solid, Shell, Face, Edge, Wire, Vertex
truck-geometry  = "0.6"   # BSpline, NURBS, Plane, Sphere 等
truck-polymesh  = "0.6"   # PolygonMesh（頂点・インデックス）
truck-meshalgo  = "0.4"   # メッシュ化トレイト（MeshableShape 等）
truck-shapeops  = "0.4"   # ブール演算 and() / or()
truck-stepio    = "0.3"   # STEP 出力（入力は未実装）
```

### 主要型対応表

| opencascade-rs | truck | 備考 |
|---|---|---|
| `Shape` | `Solid<Point3, Curve, Surface>` | トポロジ+形状の複合型 |
| `opencascade::mesh::Mesh` | `PolygonMesh` | `truck-polymesh` |
| `Face` | `Face<Point3, Curve, Surface>` | |
| `Edge` | `Edge<Point3, Curve>` | |
| `DVec3` (glam) | `Point3` / `Vector3` (cgmath) | 座標系は同一（右手系 mm） |

truck-modeling の type alias を使うと型パラメータを省略できる。

## 3. ギャップ分析（未解決課題）

### 3.1 STEP 入力 ★最大の問題★

`truck-stepio` は現時点で **出力のみ** 実装されており、STEP ファイルの読み込みは未対応。

**選択肢:**

| 案 | 方法 | 難度 |
|---|---|---|
| A | `ruststep` / `iso-10303` など別クレートで STEP パース | 高（形状再構築が必要） |
| B | `step-file` クレートでパース後、truck トポロジに変換 | 高 |
| C | truck の STEP 入力実装が完成するまで OCC の `Shape::read_step()` だけ残す | 低 |
| D | 外部プロセス（FreeCAD CLI / OpenCascade Python スクリプト）で STEP → 中間形式変換 | 中 |

**当面の推奨: 案 C（部分移行）**
STEP 読み込みのみ OCC を残し、それ以外を truck に移行する。
`step_to_brep.rs` の `Shape::read_step()` → 後述の中間フォーマット変換パイプラインで OCC を限定的に使い続ける。

### 3.2 中間フォーマット（BRep → truck ネイティブ）

現在: STEP → OCC BRep バイナリ → S3 保存 → OCC BRep 読み込み

OCC BRep ファイルフォーマットは truck では読み書きできない。

**代替案:**

| 案 | フォーマット | 備考 |
|---|---|---|
| A | truck の `Serialize` → `bincode` / JSON | truck Solid は serde 対応済み。型が変わるため再生成が必要。 |
| B | STEP ファイルのまま S3 保存（中間 BRep をやめる） | STEP 読み込みが未実装なので truck 側で使えない。 |
| C | OCC BRep を truck STEP 出力で置き換え（OCC→truck型変換が必要） | 変換実装が困難。 |

**推奨: 当面は案 A（bincode による truck Solid のシリアライズ）**
- S3 キー形式は変えず `{content_hash}` のまま
- STEP → (OCC で読み込み) → (truck 型に変換) → bincode → S3
- S3 → bincode → truck Solid として使用

ただし「OCC → truck 型変換」の実装が必要になる（3.3 参照）。

### 3.3 OCC Shape → truck Solid 変換

OCC の `TopoDS_Shape` のトポロジ（頂点・辺・面）を truck のトポロジに変換する処理が必要。

**現実的な実装方針:**

1. OCC でメッシュ化（`shape.mesh_with_tolerance()`）してポリゴンメッシュを取得
2. `truck-polymesh::PolygonMesh` として保持する（B-rep ではなくメッシュとして）
3. S3 には `PolygonMesh` を bincode シリアライズして保存

この場合、下流のブール演算（`shape_stretch`）が **メッシュベースの演算** になる。
truck-shapeops は `Solid`（NURBS B-rep）を要求するため、メッシュ上でのブール演算が必要になり難度が上がる。

**より現実的な推奨方針:**

- STEPファイル取り込みは引き続き OCC を使用
- OCC で「メッシュ化まで」行い `PolygonMesh` を保存
- `shape_stretch` を truck-shapeops ではなくメッシュ変形アルゴリズムで再実装
  （切断座標でメッシュを分割→移動→隙間をポリゴンで埋める）

### 3.4 ブール演算

`truck-shapeops` が提供するのは:
- `and(a, b, tol)` → 積集合（intersection）= OCC の `intersect()`
- `or(a, b, tol)` → 和集合（union）= OCC の `union()`
- **差集合（subtract）は直接提供されない**

差集合は `A − B = A ∩ B̄` で実現可能:
```rust
// B の面の向きを反転して not B を作り、A との積集合を取る
let b_not = b.not(); // 面の向きを全反転
let result = truck_shapeops::and(&a, &b_not, tol);
```

ただし truck-shapeops の演算結果は **STEP 出力不可**（制限あり）。

### 3.5 半空間プリミティブ

`Shape::half_space(origin, normal)` は truck には存在しない。

**代替:** 十分大きなボックスで代用

```rust
// 例: X=cut_x を境界とした正側の半空間
let big = 1_000_000.0_f64;
let half = builder::tsweep(&face_at_cut_x, Vector3::new(big, 0.0, 0.0));
// または単純に大きなボックス
let half_box = primitives::cube_at(cut_x, -big, -big, big*2, big*2, big*2);
```

`shape_stretch.rs` の `stretch_axis()` は半空間で形状を分割する設計になっているため、
このロジック自体を「平面でメッシュを分割する」アルゴリズムに書き直す必要がある。

### 3.6 Face の法線・重心

`shape_stretch.rs` の `extrude_cut_faces()` は:
- `face.normal_at_center()` → 面中心の法線
- `face.center_of_mass()` → 面の重心

truck では:
```rust
// Face の表面 (Surface) を取得してパラメータ空間中心で評価
let surface = face.surface();
let (u_mid, v_mid) = surface.parameter_range().midpoint();
let point = surface.subs(u_mid, v_mid);    // 中心座標
let normal = surface.normal(u_mid, v_mid); // 法線
```

ただし、これは「パラメータ空間の中点」であり「重心」とは一致しない場合がある。
フラットな面（平面）では問題ないが、曲面ではずれる可能性がある。

### 3.7 face.extrude()

`Shape::from(face.extrude(dir))` → truck では `tsweep(face, vector)`:
```rust
let extruded_shell = builder::tsweep(&face, vector);
let extruded_solid = Solid::new(vec![extruded_shell]);
```

### 3.8 edge.approximation_segments()

`shape_to_glb.rs` でエッジの折れ線近似に使用。

truck-meshalgo の `PolylineableCurve` トレイトを実装すれば同等の処理が可能:
```rust
// PolylineableCurve::polyline(curve, tol) で近似点列を取得
let polyline = curve.polyline(tol);
```

### 3.9 shape.clean() と deep_copy()

- `shape.clean()` → truck には直接の相当物なし。ブール演算後のトポロジ修復が目的だったが、truck-shapeops はそもそもクリーンな結果を返す前提のため不要になる可能性。
- `shape.deep_copy()` → truck の型は `Clone` で対応。

## 4. ファイル別修正方針

### 4.1 `step_to_brep.rs` → `step_to_mesh.rs`（改名推奨）

**変更前（OCC）:**
```
STEP → Shape::read_step() → shape.write_brep_bin() → S3保存
```

**変更後（OCC + truck）:**
```
STEP → Shape::read_step() [OCC] → shape.mesh_with_tolerance() [OCC]
     → PolygonMesh [truck-polymesh] → bincode::serialize() → S3保存
```

- `brep_path` → `mesh_path`（`.brep` 拡張子 → `.binmesh` 等）
- S3 キー形式は変えない（`{content_hash}`）
- アップロード先の Content-Type を `application/octet-stream` のまま継続

OCC の依存はこのファイルのみに限定される（将来的な OCC 撤廃ポイント）。

### 4.2 `shape.rs` → `Shape` 型を `PolygonMesh` に変更

```rust
// 変更前
use opencascade::primitives::Shape;
pub async fn collect_shape(...) -> Result<HashMap<String, Shape>, String>

// 変更後
use truck_polymesh::PolygonMesh;
pub async fn collect_shape(...) -> Result<HashMap<String, PolygonMesh>, String>
```

BRep 読み込み:
```rust
// 変更前
Shape::read_brep_text(&tmp).or_else(|_| Shape::read_brep_bin(&tmp))

// 変更後
let bytes = std::fs::read(&tmp)?;
bincode::deserialize::<PolygonMesh>(&bytes)
```

`shape()` 関数の末尾の `occ_shape.mesh_with_tolerance()` 呼び出しは不要になる（すでにメッシュ）。
`create_glb(&mesh, &occ_shape)` → `create_glb(&mesh)` に簡略化。

### 4.3 `shape_to_glb.rs`

メッシュ型が `opencascade::mesh::Mesh` → `truck_polymesh::PolygonMesh` に変わるので、
頂点・インデックスのアクセス方法を変更:

```rust
// 変更前
let positions: Vec<[f32; 3]> = mesh.vertices.iter()
    .map(|v| [v.x as f32, v.y as f32, v.z as f32])
    .collect();
let idx_count = mesh.indices.len();

// 変更後（PolygonMesh は三角形化済みであることを前提）
let positions: Vec<[f32; 3]> = mesh.positions().iter()
    .map(|p| [p.x as f32, p.y as f32, p.z as f32])
    .collect();
// PolygonMesh は tri_faces() でインデックスを取得
```

エッジ描画（`shape.edges()` + `edge.approximation_segments()`）は **PolygonMesh には存在しない**。

**選択肢:**
1. エッジ描画機能を廃止（GLBの `edgeAccessor` extras も削除）
2. `PolygonMesh` から境界エッジ（隣接三角形が1枚のエッジ）を抽出して代用
3. OCC でエッジ情報だけ並行して取得（移行期の妥協案）

フロントエンドがエッジ描画を使っているなら選択肢2か3が必要。

### 4.4 `shape_stretch.rs` → 全面的に書き直し

`shape_stretch` の現実装はすべて OCC API に強く依存しており、truck の B-rep ブール演算で直接置き換えるのは困難（半空間の問題など）。

**移行方針: メッシュベースの stretch アルゴリズムに切り替える**

```
入力: PolygonMesh + cut座標 + delta
1. 指定軸のcut座標で三角形を分割（三角形の切断）
2. 正側の頂点を delta だけ平行移動
3. 切断面に生じたギャップをポリゴンで補間
出力: PolygonMesh
```

この「メッシュ平面切断」は `truck-meshalgo` の機能では直接提供されていないため、
独自実装またはサードパーティクレートが必要。

候補:
- [`mesh-tools`](https://crates.io/crates/mesh-tools) 等
- 独自実装（三角形と平面の交差判定→新頂点生成）

## 5. 移行ロードマップ

### フェーズ 1: 中間フォーマット変更（OCC は残す）

目標: S3 に保存するデータを OCC BRep → `PolygonMesh` bincode に変更

1. `step_to_brep.rs` を修正: STEP→OCC→メッシュ化→bincode→S3
2. `shape.rs` を修正: S3から bincode→PolygonMesh 読み込み
3. `shape_to_glb.rs` を修正: `PolygonMesh` からGLB生成
4. 既存の BRep キャッシュは無効化（S3の`{hash}`を全再生成）

### フェーズ 2: shape_stretch のメッシュ化

目標: `shape_stretch.rs` を PolygonMesh ベースに書き直す

1. 平面でのメッシュ切断アルゴリズム実装
2. 頂点移動・ギャップ補間の実装
3. OCC の `intersect()` / `subtract()` / `union()` 依存を除去

### フェーズ 3: OCC 依存の完全除去

目標: `Cargo.toml` から `opencascade` を削除

1. STEP 読み込みを別手段で実現（truck-stepio の STEP 入力実装を待つか別クレート）
2. または FreeCAD/OCCT Python スクリプトを外部プロセスとして呼び出す形を継続

## 6. リスクと懸念点

| リスク | 影響 | 対策 |
|---|---|---|
| truck-stepio の STEP 入力が長期間未実装のまま | フェーズ 3 が完了できない | OCC を STEP 読み込み専用で残す設計を固める |
| メッシュベースの stretch は B-rep ベースより品質が劣る | 出力形状の精度低下 | 許容できる精度かどうかを事前検証 |
| truck-shapeops の差集合が不安定 | Union/Subtract 演算の信頼性 | フォールバックとして OCC ブール演算を保持 |
| PolygonMesh ではブール演算の順次適用ができない | Union/Subtract/Intersect ノードが実装できない | shape_stretch 以外の演算ノードは当面 OCC を使い続ける |
| 既存の `.brep` S3 キャッシュとの互換性破壊 | 全ユーザーの変換キャッシュが無効化 | 移行時に再変換バッチを実行、またはキー体系に version prefix を追加 |

---

## 7. 結論・推奨方針

truck は**純 Rust の B-rep カーネル**として有望だが、2026年3月時点では:
- **STEP 入力が未実装**（最大の障壁）
- **ブール演算の差集合が直接未サポート**
- **半空間プリミティブがない**

これらの制約から、**完全移行は現時点では非現実的**。

現実的な段階的アプローチ:

1. **短期**: OCC を STEP 読み込み専用に限定し、その後の処理（メッシュ化・GLB 生成）を truck/独自コードに移行
2. **中期**: `shape_stretch` をメッシュベースで再実装（OCC ブール演算を排除）
3. **長期**: truck-stepio の STEP 入力実装完成後に OCC を完全排除

truck への依存追加自体は早い段階で行い、`PolygonMesh` や変換関数を部分的に使い始めることで移行コストを分散させる。

## 8. 仮定: STEP 入力が実装済み・キャッシュ形式も自由な場合の残課題

> 「truck-stepio が STEP 入力に対応し、S3 キャッシュを OCC BRep ではなく truck ネイティブ形式（例: bincode シリアライズの `Solid`）で保存できる」と仮定した場合、移行の障壁は何が残るか。

この場合パイプラインは次のように整理される：

```
STEP → truck::Solid (読み込み) → bincode → S3
S3   → bincode → truck::Solid → ブール演算 → メッシュ化 → GLB
```

以下に残る課題を重要度順に整理する。

---

### 8.1 差集合 API がない（未確認・要検証）★重要

`truck-shapeops` が提供するのは `and()`（積集合）と `or()`（和集合）のみ。

理論上は `A − B = A ∩ B̄` なので、面の向きを全反転した `B.not()` と `and()` を組み合わせれば差集合が得られるはずだが、**truck-shapeops の実装がこのパターンを正しく処理するか公式には未確認**。

```rust
// 差集合の試み（未検証）
// Solid::not() が「面の向きを全反転 = 補集合」として機能する場合:
let b_inverted = b.clone(); // face ごとに invert が必要
let result = truck_shapeops::and(&a, &b_inverted, tol);
```

`ShapeNode::Subtract` / `ShapeNode::Intersect` を実装するためには差集合が必須であり、
この点が解決できないと `shape_stretch.rs` の `shape.subtract()` 置き換えも止まる。

**対処方針:**
- truck-shapeops のソースを読んで `not()` の挙動を確認する
- 差集合のテストケース（単純なボックス同士）で動作検証してから移行着手する

---

### 8.2 truck-shapeops の成熟度・安定性★重要

`truck-shapeops` は OCC（数十年の実績）と比べて新しいライブラリであり、実用形状（STEP 由来の複雑な B-rep）でのブール演算の安定性が保証されていない。

懸念ポイント:
- 演算結果が `Option<Solid>` で返り、**失敗（`None`）が起きうる**
- 接触面・共面・極小形状などのエッジケースでの挙動が未知
- `shape_stretch` は `intersect` / `subtract` / `union` / `extrude` を組み合わせる複雑な演算チェーンであり、各ステップで失敗する可能性がある

OCC では `shape.intersect(&half).shape` のように失敗しにくい設計だったが、truck では:
```rust
let result = truck_shapeops::and(&shape, &half_box, tol)
    .ok_or("intersection failed")?; // None の場合にエラー
```
エラーハンドリングは明示的になる一方、失敗率そのものを実測で把握する必要がある。

**対処方針:**
- 実際のユーザー STEP ファイル（数十種）で `shape_stretch` のブール演算を通して合格率を測定する
- 失敗時のフォールバック戦略（未変形のまま返す等）を設計しておく

---

### 8.3 shape_stretch: 切断面 Face の検出ロジック

`extrude_cut_faces()` は切断後の `part_neg` から「法線が切断軸に垂直で、重心座標が `cut_coord` に近い Face」を探す。
これに OCC の `face.normal_at_center()` と `face.center_of_mass()` を使っている。

truck での代替:

**法線の取得 — 平面 Face の場合は可能**

```rust
// truck-geometry の Plane は normal() を持つ
// Face::surface() が Plane 型ならそのまま使える
if let Some(plane) = face.surface().as_plane() {
    let normal = plane.normal(); // Vector3
}
```

ただし truck-stepio が読み込む STEP の Face は `NurbsSurface` や `RevolutedCurve` など
平面以外の場合もある。その場合は表面をパラメータ空間中点でサンプリングして法線を計算する。

**重心の取得 — 直接 API がない**

truck は `Face::center_of_mass()` に相当する API を提供していない。
代替手段:

1. **境界ワイヤの頂点平均** — `face.boundaries()` から全 `Vertex` を取り出して座標平均。精度は低いが実用的。
2. **パラメータ空間中点のサンプリング** — `face.surface().subs(u_mid, v_mid)` で中心点を近似。平面なら正確。
3. **メッシュ化後の三角形重心平均** — 最も正確だが計算コストがかかる。

`shape_stretch` が想定する形状（機械部品の切断断面）は平面 Face のはずなので、
**パラメータ空間中点サンプリング（案2）で実用上は問題ない**。

---

### 8.4 半空間プリミティブの大きなボックス近似

`Shape::half_space(origin, normal)` は理論的に無限の半空間を表すが、
truck での代替は巨大な有限ボックスになる。

```rust
// 例: X=cut_x の正側を切り取る半空間の代替
let big = 1_000_000.0_f64; // 100万mm = 1000m
let half_box = builder::tsweep(
    &builder::tsweep(
        &builder::tsweep(&vertex(Point3::new(cut_x, -big, -big)), vec3(2.0 * big, 0.0, 0.0)),
        vec3(0.0, 2.0 * big, 0.0),
    ),
    vec3(0.0, 0.0, 2.0 * big),
);
```

**実用上の問題:** mm スケールの機械部品（最大でも数百 mm 程度）に対しては `big = 1_000_000` で十分。
ただし、ブール演算の数値精度（`tol` パラメータ）と `big` の値のバランスに注意が必要。

---

### 8.5 shape.clean() の代替がない

OCC では `BRepLib::SewShape()` + `ShapeFix_Shape` 相当の `shape.clean()` を
ブール演算チェーンの最後に呼んでいた（トポロジのほつれを修復する目的）。

truck-shapeops の出力は truck の整合性モデル上で構築されるため、
理論的には「ほつれ」は発生しないはずだが、**実際のブール演算後に `is_geometric_consistent()` が
失敗するケースが出た場合の修復手段がない**。

OCC では `clean()` でリカバリできた状況が、truck では「演算失敗（`None`）として扱う」しかなくなる可能性がある。

---

### 8.6 glam → cgmath の型変換（工数）

現在のコードは `glam::DVec3` を使用しているが、truck は `cgmath::Point3` / `cgmath::Vector3` を使う。

```rust
// 変換ボイラープレートの例（各呼び出し箇所に必要）
let truck_vec = cgmath::Vector3::new(dx, dy, dz);
let truck_pt  = cgmath::Point3::new(cx, cy, cz);
```

`shape_stretch.rs` のシグネチャ変更（`dvec3` → `cgmath::Vector3`）と、
`glam` クレートの削除（`Cargo.toml`）が必要。工数は小さいが漏れが出やすい。

---

### 8.7 メッシュ化の tolerance 指定

現在は `shape.mesh_with_tolerance(0.1)` で 0.1mm の偏差を指定している。

truck-meshalgo の `MeshableShape` / `RobustMeshableShape` トレイトは
`tolerance` に相当するパラメータを受け取れるが、**OCC の chord deviation と
truck の tolerance が同じ意味・単位かは要確認**。

GLB の見た目に直接影響するため、デフォルト値の調整が必要になる可能性がある。

---

### 8.8 残課題サマリ（STEP 入力・キャッシュ形式解決後）

| 課題 | 重要度 | 解決難度 | 備考 |
|---|---|---|---|
| 差集合 API（`not()` + `and()` の動作確認） | ★★★ | 中（要検証） | Subtract/Intersect ノードの実装ブロッカー |
| truck-shapeops の安定性（実形状での合格率） | ★★★ | 中（要計測） | shape_stretch の実運用ブロッカー |
| 切断面 Face の重心取得 | ★★ | 低（近似で対応可） | パラメータ空間中点で実用上十分 |
| 半空間の大きなボックス近似 | ★ | 低（数値調整のみ） | mm スケールなら問題なし |
| `shape.clean()` の代替なし | ★ | 低（発生率次第） | 演算失敗として処理するしかない |
| glam → cgmath 型変換 | ★ | 低（機械的作業） | 変換ミスに注意 |
| メッシュ化 tolerance の調整 | ★ | 低（目視確認） | GLB の見た目で確認 |

**結論:** STEP 入力とキャッシュ形式の問題を解決した後に残る課題は、どれも「OCC では一行だったものが truck では数行＋要検証」レベルのものが多く、アーキテクチャを壊すものではない。ただし**差集合の動作確認と truck-shapeops の安定性**だけは移行着手前に必ず実測・検証する必要がある。
